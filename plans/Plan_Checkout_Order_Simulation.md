# Plan: Checkout & Order Simulation

- **Date:** 2026-05-13
- **Status:** Revised (post AUDIT-003)
- **Author:** julian

## Objective

Implement a full purchase simulation flow — from cart to confirmed order — excluding real payment processing. The user must be able to enter a delivery address, review the order summary, confirm the purchase, and track its simulated delivery status. This closes the e-commerce loop and makes the app a realistic shopping prototype.

## Context

The cart is functional: products are listed, added to the cart, and managed via `CartContext` and the Spring Boot backend. The MySQL schema already defines `orders`, `order_items`, `order_status_history`, `payments`, and `deliveries` tables (Flyway `V1__init.sql`), but no backend services, controllers, or frontend pages exist for them yet. RabbitMQ and Kafka are both configured and available for async processing.

Key schema details that drive implementation decisions:
- `orders.delivery_address_json` is a **JSON column** — address is not relational; requires a backend DTO and a JPA JSON converter.
- `orders.order_number VARCHAR(50) NOT NULL UNIQUE` — must be generated by the application.
- `orders.business_id NOT NULL` — sourced from the active cart's `business_id`.
- `deliveries.status` is a separate column from `orders.status` — two independent enums are needed.
- `payments` has required fields (`provider`, `status`) with no DB default — a stub record must be inserted at checkout time.

## Technology & Methodology Recommendations

| Layer | Recommendation | Reason |
|-------|---------------|--------|
| Backend entities | Map existing DB schema with Spring Data JPA | Tables are already defined; avoids schema drift |
| Order flow | Synchronous REST for checkout confirmation | Predictable UX; errors are surfaced immediately |
| Post-order events | RabbitMQ (already wired) | Decouples delivery simulation from the HTTP response |
| Delivery simulation | Spring `@Scheduled` task with status/time filtering | Advances order status automatically without a real courier API |
| Frontend routing | React Router (already used) | Multi-step checkout as sequential routes |
| Form validation | React Hook Form + Zod | Type-safe, integrates cleanly with TypeScript |
| Checkout state | New `CheckoutContext` | Holds address and order draft across the 3-step flow; mirrors `CartContext` pattern |
| Order history | New protected page + `/api/orders` endpoint | Mirrors the existing catalog/cart pattern |

## Steps / Phases

### Phase 1 — Backend: Order domain (entities, repositories, types)
1. Create JPA entities mapping the existing schema: `Order`, `OrderItem`, `OrderStatusHistory`, `Delivery`.
2. Define `AddressDTO` record with fields: `street`, `city`, `postalCode`, `country`. Add a `@Convert` JPA converter that serializes/deserializes this record to/from the `delivery_address_json` JSON column.
3. Define two status enums:
   - `OrderStatus`: `PENDING → CONFIRMED → PREPARING → SHIPPED → DELIVERED → CANCELLED`
   - `DeliveryStatus`: `PREPARING → SHIPPED → DELIVERED`
4. Define `order_number` generation strategy: `"ORD-" + year + "-" + first 8 chars of a random UUID` (e.g., `ORD-2026-a3f7c91b`). Implement as a static utility method.
5. Create Spring Data repositories for each entity.

**Success:** entities persist correctly via a repository unit test or Swagger call.

### Phase 2 — Backend: Checkout service
1. `CheckoutService.placeOrder(userId, addressDTO)` — all steps inside a single `@Transactional` method:
   a. Load the active cart and its items; read `cart.businessId` as the order's `business_id`.
   b. Calculate totals:
      - `subtotal = Σ(unit_price × quantity)` per cart item.
      - `tax_total = Σ(unit_price × quantity × vat_rate)` per item.
      - `shipping_fee = 0.00` (simulated free shipping).
      - `total = subtotal + tax_total + shipping_fee`.
   c. Generate `order_number` using the strategy from Phase 1.
   d. Persist `Order` with status `CONFIRMED` and `delivery_address_json = addressDTO`.
   e. Snapshot each cart item into `order_items` (capturing `product_name`, `sku`, `unit_price`, `vat_rate`, `line_total`).
   f. Insert a `Payment` stub: `provider = "SIMULATED"`, `status = "SIMULATED"`, `amount = total`, `paid_at = now()`.
   g. Decrement inventory stock for each product (with optimistic locking — see risks).
   h. Deactivate the cart (`is_active = false`).
   i. Write the initial `order_status_history` row: `from_status = null`, `to_status = CONFIRMED`.
2. Publish `OrderPlacedEvent` to RabbitMQ after the transaction commits (use `TransactionSynchronizationManager` or a `@TransactionalEventListener`).
3. Add `OrderService.getOrdersByUser(userId)` and `getOrderById(orderId, userId)` as read methods (note: `userId` is always passed to enforce ownership).

**Success:** a POST to `/api/orders/checkout` creates a DB record, reduces stock, inserts a payment stub, deactivates the cart, and publishes an event.

### Phase 3 — Backend: Order controller & API
1. Define DTOs: `CheckoutRequestDTO { addressDTO }`, `OrderSummaryDTO`, `OrderDetailDTO` (never expose JPA entities directly).
2. `POST /api/orders/checkout` — body: `CheckoutRequestDTO` → returns `OrderSummaryDTO` with order number and total.
3. `GET /api/orders` — returns list of `OrderSummaryDTO` for the authenticated user.
4. `GET /api/orders/{id}` — returns `OrderDetailDTO` with items, address, status timeline.
   - **Authorization:** service must verify `order.userId == principal.userId`; throw `403 Forbidden` otherwise. This prevents IDOR attacks.
5. Secure all endpoints with the existing JWT filter (role `USER`).

**Success:** Swagger UI shows all endpoints; attempting to access another user's order returns 403.

### Phase 4 — Backend: Delivery simulation
1. `DeliverySimulatorService` listens to the RabbitMQ `OrderPlacedEvent`.
2. Creates a `Delivery` record with `status = PREPARING`, `created_at = now()`, `eta_from = now() + 2 min`, `eta_to = now() + 6 min`.
3. A `@Scheduled` task runs every 60 seconds and:
   - Queries orders with `status IN (CONFIRMED, PREPARING, SHIPPED)`.
   - For each, checks whether the minimum simulated time in the current state has elapsed (uses `delivery.eta_from` as the threshold).
   - If elapsed, advances both `order.status` and `delivery.status` by one step.
   - Writes a row to `order_status_history` per transition.
   - This logic is idempotent — re-running on already-advanced orders is a no-op.
4. State progression:
   - `CONFIRMED → PREPARING` (immediate, on delivery record creation)
   - `PREPARING → SHIPPED` (after `eta_from` elapsed)
   - `SHIPPED → DELIVERED` (after a second fixed delay, e.g., `eta_from + 4 min`)

**Success:** placing an order and waiting ~6 minutes triggers automatic status progression visible via `GET /api/orders/{id}`.

### Phase 5 — Frontend: Multi-step checkout flow
Introduce a `CheckoutContext` (analogous to `CartContext`) to hold address and order state across steps.

Three sequential routes under `/checkout`:

| Step | Route | Content |
|------|-------|---------|
| 1 | `/checkout/address` | Address form (React Hook Form + Zod): `street`, `city`, `postalCode`, `country` |
| 2 | `/checkout/summary` | Read-only summary: cart items, subtotal, VAT, shipping (€0.00), total |
| 3 | `/checkout/confirmation` | Order number, estimated delivery, "Go to my orders" CTA |

- "Proceed to checkout" button on `CartPage` navigates to step 1.
- **Empty cart guard:** on entry to `/checkout/address`, redirect to `/cart` if cart is empty.
- Address submitted in step 1 is stored in `CheckoutContext`; the summary in step 2 is computed client-side from cart items.
- Confirming in step 2 calls `POST /api/orders/checkout` with the stored address; on success, stores the returned order in context and navigates to step 3.
- On reaching step 3, `CartContext.clearCart()` is called.
- All checkout routes are protected by `PrivateRoute`.
- Browser back/forward within the flow is handled by the context; navigating back past step 1 clears the context.

**Success:** a user can complete the 3-step flow, see a confirmation with the real order number, and find the cart empty afterward.

### Phase 6 — Frontend: Order history & detail
1. New page `OrdersPage` at `/orders` — lists all past orders with status badge; polling not needed (refresh on page load is sufficient for a prototype).
2. New page `OrderDetailPage` at `/orders/:id` — shows items, delivery address, and status timeline from `order_status_history`.
3. Link in the `Navbar` for authenticated users.
4. New API module `src/api/orders.ts` mirroring the existing `cart.ts` pattern.

**Success:** a user can view their order list and individual order detail with the full status timeline.

## Dependencies and Risks

| Item | Detail |
|------|--------|
| Existing schema | `V1__init.sql` must be accurate; verify column names before writing entities |
| Inventory race condition | Two users checking out the last item simultaneously — use `@Version` on `Inventory` entity (optimistic locking); catch `OptimisticLockException` and return HTTP 409 with a user-friendly message |
| RabbitMQ availability | Docker Compose must be running; add a dead-letter queue so failed `OrderPlacedEvent` messages are not silently dropped |
| Partial transaction failure | If RabbitMQ publish fails after DB commit, delivery simulation never starts — use `@TransactionalEventListener(phase = AFTER_COMMIT)` so the event is published only after the transaction succeeds |
| Scheduler idempotency | `@Scheduled` task must filter by status + elapsed time to avoid double-advancing orders on app restart |
| IDOR on order endpoints | `GET /api/orders/{id}` must verify ownership — implemented as a service-layer check, not only at the controller level |
| Address JSON contract | Frontend form field names must match the backend `AddressDTO` exactly — define the DTO first |
| Guest users | Checkout is restricted to authenticated users; guest cart → `/login?redirect=/checkout/address` |

## Success Criteria

- A logged-in user can complete the full flow: cart → address → summary → confirmation.
- The order appears in the DB with correct items, prices, payment stub, and status `CONFIRMED`.
- Inventory stock is decremented after checkout; concurrent checkout of the last item returns HTTP 409 for the second user.
- Order status advances automatically from `CONFIRMED` to `DELIVERED` without manual intervention within ~6 minutes.
- Attempting to access another user's order returns HTTP 403.
- The user can view their order history and individual order detail with the status timeline.
- The cart is empty after a successful checkout.
- Navigating directly to `/checkout/address` with an empty cart redirects to `/cart`.

## Additional Notes

- Polling vs. real-time: order status refresh via page reload is intentional for this prototype; WebSocket/SSE is out of scope.
- Reviews: the `reviews` table is also in the schema — a post-delivery review flow is a natural Phase 7 but is out of scope here.
